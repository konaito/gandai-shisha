## 1. Build in Publicとは何か

### 定義

Build in Public（BIP）とは、事業やプロダクトの構築過程を、完成を待たずにリアルタイムで公開する手法。計画・調査・意思決定・失敗・数字——すべてを透明にすることで、観客を巻き込みながら事業を育てる。

従来のビジネスでは「完成してから発表」が常識だった。BIPはその逆を行く。

| 従来のアプローチ | Build in Public |
|----------------|-----------------|
| 完成品を発表 | 未完成の状態から公開 |
| 成功だけを見せる | 失敗も数字も全部見せる |
| 情報は競争優位 | 情報公開が競争優位 |
| 顧客は「買う人」 | フォロワーは「一緒に作る人」 |

### BIPの本質

BIPは単なるマーケティング手法ではない。**事業構築のプロセス自体をコンテンツにする**ことで、以下の好循環を生む:

1. プロセスを公開する → 注目が集まる
2. 注目が集まる → フィードバックが得られる
3. フィードバックが得られる → 事業が改善される
4. 改善される → また公開する → さらに注目が集まる

## 2. 海外の成功事例

### Pieter Levels（@levelsio）

- **何をした**: NomadList, RemoteOK, PhotoAI 等を一人で作り、収益をリアルタイムで公開
- **BIPの方法**: Twitterで毎日の売上・MRR・失敗を投稿。コードの進捗もスクリーンショットで共有
- **結果**: フォロワー40万人以上。プロダクトを出すたびに即座にユーザーがつく
- **学び**: 「完璧を目指す前に出す」「数字を隠さない」「毎日更新する」

### Indie Hackersコミュニティ

- **概要**: ソロファウンダーが売上・失敗・学びを共有するコミュニティ（Stripeが運営）
- **文化**: 月次の収益レポートを公開する文化が根付いている
- **学び**: BIPは一人でやるものではない。同じ志の仲間とのコミュニティがあると継続しやすい

### Buffer

- **何をした**: SNS管理ツールの会社が、全社員の給与・収益・意思決定プロセスを公開
- **BIPの方法**: 公式ブログで「Open Blog」を運営。採用プロセスも公開
- **結果**: 透明性がブランドの核になり、競合との差別化に成功
- **学び**: BIPは個人だけでなく、組織のブランディングにも使える

### Pat Walls（Starter Story）

- **何をした**: スモールビジネスの成功事例を集めるメディアを、自身もBIPで成長させた
- **BIPの方法**: 毎月の売上・トラフィック・失敗を詳細にブログで公開
- **結果**: 年間100万ドル以上の収益。BIPコンテンツ自体が集客装置に

## 3. 日本での実践例

### 日本のBIP事例

日本ではBIPはまだ主流ではないが、先駆者は存在する。

| 事例 | 概要 |
|------|------|
| **けんすう（アル株式会社）** | 新サービスの構想段階からnoteで公開。ユーザーの意見を取り込みながら開発 |
| **yutori（アパレル）** | 古着ブランドの立ち上げ過程をSNSで公開。Z世代の共感を獲得 |
| **個人開発者コミュニティ** | Xで月次の売上報告をする文化が日本の個人開発者にも広がりつつある |

### 日本でBIPが少ない理由

- **失敗を見せる文化がない**: 日本では「恥」の文化が強く、失敗を公開することへの心理的抵抗が大きい
- **数字を出すのが怖い**: 売上や収益を公開することへの抵抗感
- **パクられる恐怖**: アイデアや戦略を公開すると真似されるという不安
- **完璧主義**: 「未完成のものを見せるのは恥ずかしい」という意識

### 逆に言えば

日本でBIPをやる人が少ないからこそ、**やるだけで差別化になる**。特に地方都市（盛岡）では、事業構築の過程を全公開している人はほぼ皆無。

## 4. メリット・デメリット

### メリット

| メリット | BiPシーシャでの具体例 |
|---------|---------------------|
| **開業前から認知を獲得** | サイトを公開した時点で、まだ物件も決まっていないのに話題になる |
| **フィードバックが得られる** | 料金設計を公開 → 「高い」「安い」の声が直接聞ける |
| **信頼の蓄積** | プロセスを見てきた人は、オープン時に「最初から見てたから行きたい」となる |
| **メディア露出** | 「大学生がAIと一緒にシーシャ屋を作る過程を全公開」は記事になりやすい |
| **自分の学びが資産になる** | 調べたことがドキュメントになり、サイトのコンテンツになる |
| **アカウンタビリティ** | 公開することで「やらなきゃ」というプレッシャーが生まれ、継続力UP |
| **コミュニティの形成** | 応援してくれる人が自然と集まる |

### デメリット

| デメリット | 対策 |
|-----------|------|
| **情報を真似される** | シーシャ店の差別化は「場所」と「コミュニティ」。情報だけ真似しても同じ体験は作れない |
| **失敗が公になる** | 失敗を「学び」として言語化する。むしろ失敗コンテンツのほうがエンゲージメントが高い |
| **プライバシーの線引き** | 公開する情報と非公開の情報を事前に決めておく（後述） |
| **更新のプレッシャー** | 完璧な更新を目指さない。「今日はこれだけ」でもOK |
| **批判・否定的な反応** | BIPは「見せる」ことが目的であり、全員に好かれる必要はない |

## 5. BiPシーシャのBiP戦略

### BiPシーシャの独自性

BiPシーシャのBIPは、一般的なBIPといくつかの点で異なる。

1. **AIとの協業が丸見え**: 物件探し、料金設計、原価計算、法務確認、Webサイト自体——すべてAIとの対話で作った過程がコンテンツ
2. **地方都市での実践**: 東京やシリコンバレーではなく、盛岡という地方都市でBIPを行う希少性
3. **学生起業のリアル**: 大学3年生が一人で事業を立ち上げるプロセスの生々しさ
4. **Webサイトが主戦場**: SNSのフロー型コンテンツだけでなく、構造化されたWebサイト（shisha.konaito.com）がBIPの中心

### BiPシーシャのBIP原則

1. **数字を隠さない**: 原価率、家賃、初期投資額、売上シミュレーション——すべてサイトに載せる
2. **意思決定の根拠を見せる**: 「なぜこの物件を選んだか」「なぜこの料金にしたか」の理由を明示
3. **失敗もリアルタイムで公開**: 物件に断られた、許認可で想定外の問題が出た——そういうネガティブ情報こそ価値がある
4. **AIの使い方自体がコンテンツ**: 「AIとこうやって事業を作る」というプロセスが、シーシャに興味がない人にも刺さる
5. **更新頻度 > 完成度**: 完璧な記事を月1回より、ラフな更新を週2〜3回

### ターゲット別のBIPコンテンツ

| ターゲット | 刺さるBIPコンテンツ |
|-----------|-------------------|
| **岩手大学の友達** | 「内藤がなんかやってる」→ 応援 → 来店 |
| **シーシャ好きの学生** | 料金・場所・コンセプトの具体情報 |
| **起業に興味がある学生** | 事業計画、収支計算、失敗談 |
| **テック系・AI系の人** | AIとの協業プロセス、Claude Codeの使い方 |
| **地方のメディア** | 「大学生×AI×シーシャ×Build in Public」のストーリー |

| 指標 | 目標 | 測定方法 |
